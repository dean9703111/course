# 起點：AI 很快，但用完就丟

> Vibe Coding 的產出很快，但如果每次都在重來，效率其實是假的

## 為什麼不能只停在「生成」

### 🎯 這支影片要解決的問題
- 很多人玩 Vibe Coding，都停在一個很可惜的階段
- AI 做得出來，而且做得很快，**但一用完就丟掉了**
- 看起來效率很高，其實每一次都在重來

### 💡 真正想分享的事
- 不是「怎麼做出一個講義網站」
- 而是怎麼把一份 AI 生成的 HTML，整理成**可長期使用、維護、複用的講義框架**
- 這套框架也整理成可直接體驗的版本（repo 在資訊欄）

> **把 AI 生成結果，變成可複用的系統**
> 做得出來不是最難的，難的是它下次還是不是你的。Vibe Coding 真正的價值，在於你能不能把產出累積成自己的資產。

# 第一版：AI 生成 HTML，但問題才正要開始

> 第一版完成度很高，但「看起來完整」不代表「適合修改」

## AI 幫你做第一版

### ⚡ 一開始做的事很簡單
- 把已經寫好的課程大綱直接交給 AI
- 大綱裡有主題、段落、重點、目標
- 不做 PowerPoint，直接請 AI 生成 HTML 網頁版講義

### 📊 第一版結果
- AI 很快就做出來了
- 第一版完成度比預期還高，畫面、節奏、結構七八成像樣
- 如果只要「看起來可以用」的網頁，到這步就差不多了

## 撞牆：改不動

### 🚧 Vibe Coding 遲早撞到的問題
- AI 會自己增減文字、調整段落、改變強調方式
- 這本身不可怕，AI 本來就會這樣
- **真正麻煩的是：想修的時候，得回去改 HTML**

### 🎯 講義該維護的是什麼？
- 講義真正要被維護的是內容、邏輯、節奏
- 不是在 HTML 裡面找區塊慢慢修
- 如果要讓這件事變成工作流，不能只停留在 AI 幫你生成第一版

> **看起來完整，不代表適合修改**
> 講義要維護的是內容，不是 HTML。如果你想讓 Vibe Coding 成為工作流，就不能只停留在 AI 幫你生成第一版。

# 逆向工程：從 HTML 拿回內容主控權

> 把 HTML 逆向回 Markdown，讓內容存在可編輯、可維護、可理解的格式裡

## 關鍵決定：逆向回 Markdown

### 📝 重新思考講義的本質
- 標題、章節、次標題、程式碼、步驟、圖片、補充說明
- 這些東西本來就比較接近 Markdown 的世界
- 真正想掌控的是「內容」，就該讓內容存在**可編輯、可維護、可理解**的格式裡

### 🔄 逆向拆解的過程
- 不是一直叫 AI 幫忙修 HTML
- 而是請 AI 拆解畫面上的元素，逆向整理成 Markdown 可對應的模組

[flow]
1. AI 生成 HTML 第一版
2. 拆解畫面元素，逆向回 Markdown
3. 建立 Build 流程：Markdown → HTML
4. 內容主控權回到自己手上
[/flow]

## 建立 Build 流程

### 🏗️ 新的工作方式
- 我負責寫 Markdown
- 框架負責把 Markdown 重新 build 成最終的 HTML 網頁
- 改字、改段落、改順序、改節奏，只要改 Markdown
- HTML 只是最後的輸出結果，不再是平常要碰的主體

> **我真正想拿回來的，是內容的主控權**
> 做到這裡之後，才真正覺得不是在做一份 AI 幫忙生成的講義，而是在做一個可以重複使用的框架。

# 框架實戰：不只是頁面，而是整條講義工作流

> 進度條、深淺模式、PDF 輸出 — 優化的是準備內容、現場講課、課後交付的完整流程

## 教學節奏控制

### 📊 側邊欄與進度條
- 側邊欄不希望永遠占著畫面，收起來後保留一條進度條
- 進度條看起來只是小細節，但對講者非常有用
- 講課時直覺知道大概講到哪裡，依現場時間與狀態調整節奏

## 場地適應

### 🌗 深色與淺色模式
- 不同單位場地差很多：明亮會議室 vs 暗場演講廳
- 明亮場地用 light mode，暗場投影切 dark mode
- 不是為了功能多，而是真的上場講課就會知道，這些細節影響觀眾閱讀感受

## 課後交付

### 📄 PDF 輸出與字體控制
- 有些單位希望課後提供 PDF 講義
- 一開始就考慮 PDF 輸出時的排版
- 字體大小做成按鈕，不只影響網頁閱讀，也影響最後 PDF 呈現

> **優化的不只是頁面，而是整條工作流**
> 準備內容、現場講課、課後交付 — 這套框架真正有價值的地方，是它不是做完一次就丟的東西，而是一套之後還能繼續長大的基礎。

# 深水區：Vibe Coding 的限制與專業性

> 你想要主控權，就得承擔客製化背後的複雜度

## 不是打一行 Prompt 就能變成產品

### ⚠️ 中間踩過的坑
- 超連結沒有正確顯示
- 進度條呈現跟預期不同
- 某些區塊在 build 之後結構跑掉
- 這些 bug 最後不會有人幫你收尾，得自己修

### 🎯 客製化的代價與回報
- **想要主控權，就得承擔客製化背後的複雜度**
- 一旦跨過這一層，就能做出高度個人化的東西
- 例如：彈窗在特定時機展開、提示只在特定段落出現
- 制式工具很難做到這麼貼

> **Vibe Coding 不會自動把 Demo 變成產品**
> AI 有時候不是真的懂你要什麼，它只是很像懂。中間的 bug 和調整，需要你自己的專業判斷來收尾。

## 降低門檻：Agent Skill

### 🤖 不用背語法，讓 AI 判斷
- 把特殊 Markdown 規則整理成 Agent Skill
- 不需要把每個特殊指令都背起來
- 更重要的工作是把你真正想講的內容整理好
- 讓 AI 根據 Skill 判斷適合哪個模組、版型、結構

> **把心力放在內容打磨，而不是成為框架操作員**
> 你不需要記住所有語法細節，讓 AI 和 Skill 處理格式問題，你專注在真正重要的事：內容本身。

## 走向真正可複用的系統

### 🧱 結構化設定的思維

[flow]
1. 哪些內容放在 Markdown — 講義本體
2. 哪些資訊抽進 Config — 作者、頁尾、共通設定
3. 哪些行為交給 Skill — 格式轉換、模組判斷
4. 哪些設定用 YAML 控制 — OG、SEO、主題色
[/flow]
