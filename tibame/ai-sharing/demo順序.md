## 不要讓 AI 的「快」，變成未來的「債」

這邊先跟聊聊用 AI 寫程式最常遇到的痛點

首先是穩定性，有時我們請 AI 解決目前的問題，但改完後發現過去正常的功能被改壞了；其實我覺得改壞不是最可怕的，是你不知道他被改壞了

再來是複雜度，只要有在真實環境被使用的系統，功能一定會隨著使用者實際的需求不斷迭代；一開始我們還可以用人工的方式確認流程的正確性，但隨著使用的複雜度上升，人類是記不住，也沒有時間把所有操作路線測試一遍的

最後是擴充性，當系統的架構逐漸複雜，如果沒有對應的規格文件、自動化測試，那麼任何修改都是在走鋼索，因為一個函式有可能被多個功能共用，如果沒注意到耦合性，那新增一個功能把十個功能搞壞也是很常見的

當然，業界面對這類有可能把過去功能改壞的不確定性，還有一種常見的做法，就是直接複製貼上一份，各走各的路；但這樣當下的問題是解決了，當功能要修改時，你複製幾個地方，就有幾個地方要修改，會導致專案幾乎沒有維護性

如果我剛剛提到的這些能引起你的共鳴，可以用 123 把遇到的痛點打在留言區，互動一下，讓我知道是不是所有人都深受其害

其實準備這場分享會時，我有先看大家填寫的問卷，沒想到大部分朋友的工作年資都超過 7 年

其實早在 AI 流行前，這些問題就已經是業界的痛點；但是因為 AI 寫程式的速度太快，所以問題變得更加嚴重

### 將 AI 導入工作流

所以接下來，我就要跟大家分享，如何將 AI 導入現有工作流

在介紹的過程中，可能大家會發現，誒！這算是 AI 工具嗎？
對！其實有些事情我們未必全部都要交給 AI 來做，畢竟讓 AI 做事需要花 Token，但有些任務設計成自動化流程就好

第一個要介紹的是「Lint」，讓 AI 寫程式時，他時常會有程式碼風格不一致的問題，而且迭代新功能的過程，也常常會遺留舊的程式碼、沒有使用到的變數；而這些有問題的資訊，不管是人類還是 AI，在未來看到的時候都會感到疑惑，而 Lint 就是用來解決這個問題的，後續也會有 Demo 示範給大家看

接下來是 OpenSpec，儘管我們可以用一句話生成系統，但就算 AI 真的完成了，你對系統的掌控力還是近乎於零；所以我們會需要使用 OpenSpec 來生成對應的專案文件，他可以幫助我們知道系統有哪些功能，以及迭代的過程，這些文件不僅讓 AI 有做事的依據，也讓人能夠了解 AI 做了哪些事

然後要分享最近超夯的 Agent Skills，其實剛剛介紹的 OpenSpec 也是透過 Agent Skills 來驅動的；你可以把 Agent Skills 想像成一個技能包，安裝好後 AI 就知道該怎麼做事，但如果市面上沒有，或是不符合自己的工作流，我們就可以自己客製化，像我就針對開發常用到的 commit 訊息、branch 名稱、Pull Request 草稿進行專門的設計

另外要分享很多專案都會忽視的一點，測試程式，前面提到許多用 AI 開發會遇到的痛點，其實在導入測試後都能夠獲得很大程度地解決，讓專案新增、調整功能時更有底氣

同時我也會在課程中分享開發的基礎知識，如果說明會中對一些名詞感到陌生不需要害怕，我都會搭配具體的案例用白話的方式讓大家理解，比如為什麼要加入版本控制，這是為了讓專案出包的時候有回頭路；為何要設計分支策略，這是要區分給別人用的穩定版本，以及自己開發測試用的版本

最後會介紹 CI/CD 的自動化工作流，讓更新時都會檢查格式、測試功能；並且說明如何設計保護重點 branch 分支的策略，避免有問題的程式合併到主線

# 了解 AI 的能力範圍，而不是盲從

雖然前面分享了 AI Agent 的優勢，但並不代表用對工具就獲得無敵星星。

就像我在滑社群媒體的時候，有時都在自我懷疑，因為上面每個人感覺都是 AI 大神；隨便下幾個指令，讓一堆 AI Agent 幫自己做事就能完成複雜系統。

但我自己實測下來卻處處碰壁，作品沒做出來，錢倒是燒了不少。

## 網路上的 Demo 都是精心挑選過的

## 一堆 AI Agent 同時執行，未必會更好

另外社群最常看見的，就是有人開了一堆終端機，然後畫面狂閃，感覺每個 AI Agent 都非常努力工作

我得說一句實話，一堆 AI 同時執行，問題不一定能解決，但 Token 額度消耗絕對很快

而且現在 AI 寫程式的速度已經夠快了，按造順序執行任務，已經可以滿足絕大多數的開發情境，你也會對專案有更高的掌握度。

在工作中導入 AI，不代表全交給 AI

把 AI 能做到的事跟人必須負責的判斷分清楚，我覺得很重要；好的結果，不該靠消耗 Token 拼運氣；而是靠清楚的方向、可重複的工作流、以及人類在關鍵節點的決策。

## 為什麼需要 OpenSpec？

這邊我們會使用到 OpenSpec 這款工具

過去我們可能靠一個指令就生成系統，但這個系統具體有哪些功能，使用哪些技術，其實我們是完全沒有掌控力的

所以我們要借助 OpenSpec 這款工具來幫我們釐清專案的細節

他不僅能讓 AI 執行時更有方向，我們也可以在 AI 執行前，確認功能到底是不是我們需要的

而且他做的每件事都會留下規格文件，這樣就算同事不在，也有對應的文件可以知道過去做了哪些事情，不會有知識斷層的狀況發生

OpenSpec 會按造下面的步驟來建立規格文件與任務，這邊我就把視角切換到專案，大家會更了解他每個步驟做了哪些事

這邊因為 OpenSpec 建立會需要一段時間，所以我是直接拿完成的成果向大家展示

首先，他會建立 Proposal，這是用來確認目標與範圍的文件

你的指令可能只有一個大方向，但 AI 會把它具體並且細節化，你確認沒問題後

他才會進入 Design 文件的設計，這是用來確認使用的技術，與專案對應的風險評估

等你覺得都可行後，他就會根據功能來建立對應的 Spec，這點我覺得很重要，因為是分功能建立，所以瀏覽時更方便，如果都在寫一個文件，那讀起來太費力了

而在最後，則是建立 Task 任務清單；一開始這些欄位都是空的，AI 在完成後才會打勾

這招很聰明，可以盡可能避免 AI 偷懶不做事的狀況發生

以我現在展示的這個專案來說，有接近 100 個任務，Claude Code 大約花了 30 分鐘才完成初版的建立

# 專案協作:建立適合的 Agent Skills

> 這其實也是在解決跨專案協作的問題，每個專案的文件是散落的，Agent Skill 漸進式揭露的功能，有需要時，再去讀取

## 我在公司怎麼推：用 Workshop 把「個人習慣」變成「團隊資產」

我在公司推動 **Cursor** 導入時，開過一場線上 Workshop：讓同事理解編輯器能帶來哪些幫助（**Rules、限制文件範圍、Code Review、生成 Test Cases** 等）。  
後來 **Agent Skills** 推出後，我又開了一場 Workshop，讓大家理解如何從公司 / 部門 / 專案層級設計 Skills；現在也維護一個共用的 **Agent Skills Repository**，依情境與層級取用。

我強調的一直不是「AI 讓我寫得多快」，而是：**哪些重複、無聊、但又會影響協作品質的事，值得交給 AI**——例如：

- 需求規格草稿、測試案例整理  
- Pull Request 描述、Commit 前 Code Review 的檢查清單  
- 分段 Commit 訊息、Branch 命名規範  

這些事很多人不擅長、也不想做，但它們會直接影響 **Review 效率** 與 **專案可維護性**。**既然人不想做，那就設計成 AI 能做、而且做得一致。**

> **在請 AI 幫忙生 PR 草稿之前，我早就先在 GitLab 用 Template 把「團隊要寫什麼」講清楚。**  
> AI 不是魔法；它更像加速器：**你先有協作意識與模板，AI 才有辦法把品質放大。**

還有一個很務實的例子：教育訓練影片整理。以前 3 小時的內容要整理成可用文件很痛苦；現在我會用 AI **自動轉 SRT**、再 **標註段落重點**。這種痛點常常「做起來不難，但一直沒人做」——**只要你做了，貢獻通常會被記得很久。**

### 「槍打出頭鳥怎麼辦？」

首先，我在課程裡分享的，多半是我已在職場驗證過的路線。  
其次，**方向如果真的走偏，組織通常會比你更急著修正**；而在很多團隊裡，願意站出來把流程推動起來的人反而稀缺——你反而更可能換到資源與關注，而不是被「默默消失」。

如果你公司有 **one-on-one**，那其實就是把課堂所學「每週拿去公司試錯與對齊」的最佳場景。  
多數公司都想導入 AI，但真的知道怎麼落地的人很少；**當你成為那個能把方法講清楚、把流程跑起來的人，話語權通常會跟著來。** 推動時也更有「名份」：你可以用主管／老闆在乎的語言，去換到時間、人力與制度上的支持。

## 被看見的方式：圍繞痛點，而不是圍繞你心裡的分數

很多人會委屈：「我很努力、最難的我也扛了，為什麼主管不提拔我？」  
很多時候不是你不夠強，而是 **你沒有把成果，翻成團隊聽得懂、也覺得重要的語言**。

**世界沒有義務認識你；世界只會記得你能帶來什麼改變。**  
所以除了 AI 技術本身，我也會談一件同樣現實的事：**怎麼用市場／組織能接受的敘事，把你的工作變成「可感知的價值」。**

> 我後來能接到不少企業內訓、也能開線上課、大家願意來聽分享——本質上靠的是 **口碑**。我自媒體經營超過 6 年、寫了超過 500 篇文章，也拍了很多 Vibe Coding 教學；**那些長期累積的「可被驗證的輸出」，最後才會變成別人願意給你的機會。**  
> 不管最後報名人數多少，我都會在能力範圍內把交付做到最好——**因為口碑這種東西，只會被作品與結果記住。**

如果你對更完整的實作路徑有興趣，歡迎報名線上課程。我是鼎淵，我們，下班有約。

> **賣課不丟人，賣沒價值的課才丟人。** 我也會用同一套標準要求自己的內容品質。

這堂課想帶你從不同視角去用 AI：**做出不同選擇、累積不同槓桿，最後就會拿到不同的成果。**
