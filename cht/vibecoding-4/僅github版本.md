在團隊合作時，如果想把自己的修改合併回主線，會先開一個 Pull Request，讓其他人幫你檢查程式碼、討論是否需要修改，確認沒問題後再合併。

### 確認 Git 版本

首先，我們要確認電腦的 Git 版本

並提醒一下大家，因為這隻影片會使用到 Node.js 與 Git 等技術，如果你是第一次接觸，可以參考上方資訊卡的連結，我有專門錄一隻影片分享詳細的操作步驟喔！

安裝好 Git 後，點擊上方的 Toggle Panel 開啟終端機，輸入指令 `git -v` 確認有正確安裝

另外提醒安裝過的朋友，如果你的版本低於 2.30，建議升級到最新版本，這樣 Git Worktree 功能會較為完整

### 安裝 pnpm

另外，因為 Git Worktree 會建立多個工作目錄

而 Node.js 專案的 node_modules 動輒上百 MB

如果用 `npm` 安裝會消耗大量硬碟資源，因此我推薦大家使用 `pnpm` 這款套件

在終端機輸入 `npm install -g pnpm` 便可完成全局安裝

pnpm 會將所有套件存放在電腦的一個共用儲存區，各專案的 node_modules 只是透過連結指向同一份套件

雖然每個 Worktree 需要執行 pnpm install，但不會重新下載套件，非常適合搭配 Git Worktree 進行並行開發。

### 安裝 Extensions

然後我們把視角轉到左側選單，點擊「Extensions」的分頁，這邊推薦大家安裝兩款外掛

第一款是「Git Worktree Manager」，他能讓你透過介面輕鬆的建立、切換、清理 Worktree

第二款是「GitLens」，安裝好後，你把滑鼠移到對應的程式碼位置時，可以直接看到 Commit 的歷史

不過這邊要提醒一下大家，這些外掛大部分是由第三方開發的，安裝前請自行評估安全性

### 了解 Git Worktree 工作模式

做好前置作業後，我們先在終端機輸入 `pnpm install` 安裝套件

安裝完成後，輸入 `pnpm run dev` 便可啟動專案，下面就會有一個「localhost:3000」的網址

點擊確認專案可以順利啟動後，我們回到 Antigravity，點擊左側剛剛安裝好的「Git Worktree Manager」

然後點擊這個資料夾的符號來新增 Worktree

因為我們要從現在的主分支，切換到其他分支，所以選擇第一個「Create new branch」

至於分支的名稱就打 `test`，主要是先讓大家了解 Git Worktree 概念

至於目錄，使用系統自動生成的就好了，最後點擊「Create」，就會直接把工作區的內容複製一份過去

此時，你就會看到 test 出現在 Worktree List 當中

點擊後便會開啟一個新的工作區，打開終端機後，要在輸入 `pnpm install` 來安裝套件

你會發現套件幾乎在瞬間安裝完畢，打開 node_modules 資料夾，你會發現有箭頭圖示「↩」

代表只是建立資料夾的連結，而非重新下載套件，這就是 pnpm 的運作方式，可以幫我們節省硬碟空間

然後我們嘗試輸入 `pnpm run dev` 啟動專案，這次的網址變成了「localhost:3001」，點擊後，網頁也可以順利呈現

這邊的示範，是想要讓大家了解每個 Worktree 都是獨立的工作目錄

所以可以同時跑不同的 dev server 跟 port

當然，也代表我們可以同時指揮多個 AI Agent 並行開發不同功能

這邊也跟大家說明一下，如何把 Worktree 刪除

在刪除錢，請先把剛剛啟動的專案關閉

然後回到 Git Worktree Manager 的介面，點擊右鍵，在 Worktree Action 當中選擇 Remove Worktree，就可以移除了

## ③ 自動建立分支與開發規格

依功能設計 SPEC，拆成多個 Worktree

前面有說到，不同的功能應該使用不同的 feature branch 來開發

但具體怎麼拆分功能，以及 branch 的名稱如何命名

我們可以直接設計一個 Agent Skill 來幫我們搞定

如果你是 Fork 我的專案，在根目錄下方會有一個「.agent」的資料夾，我有放一些 Skills 讓大家跟著影片一起體驗

像這邊我設計了一個 `git-worktree-design` 技能，把 Markdown 點開來後，跟大家說明一下設計的邏輯
> 下面用截圖定格呈現
1. 首先會檢查目前專案分支的狀況，確認既有 Worktree，以及是否有沒提交的變更
2. 然後將複雜的需求拆成獨立的 feature branch 以及參考的規格後，用表格預覽，使用者確認後才會執行
3. 接下來就會根據 Skill 指定的格式建立 Worktree
4. 並順便安裝好每個 Worktree 的套件
5. 然後新增這個 feature 要執行的 SPEC，方便後續 AI Agent 的開發
6. 完成以上任務後，會再用表格像我們展示完成的結果

了解 Skill 的設計邏輯後，我們在右側與 AI 的對話視窗中，輸入指令

```

```

此時，你會發現 AI 判斷需要使用 Git Worktree 這個 Skill，並初步規劃了每個 Worktree 要參考的規格

確認沒問題後，就可以讓 AI 幫我們執行下去

稍等一段時間後，AI 說他已經幫我們完成了對應的設計

打開左側 Git Worktree Manager 的分頁，可以看到看到新增的 Worktree

這邊點擊其中一個 Worktree，你可以看到 AI 有根據我們的指示，用 pnpm 來安裝套件，所以 node_modules 下面資料夾有這個箭頭符號

並且根目錄的地方有存放給「git-Worktree-spec.md」這個給 AI Agent 參考的規格，確保執行的方向

## ④ 指揮多個 AI Agent 同時執行

但讓 AI 參考規格開發前，跟大家分享一個小技巧

先開啟與 AI 對話的視窗，點擊右上方的「...」，選擇 Workflows

我有設計一個自動化流程，這樣只要輸入快捷鍵，就可以讓 AI 知道讀取規格、執行實作、驗收結果的步驟

這邊我們先點開其中一個 Worktree，輸入「/」，選擇剛剛的那個 workflow，點擊「Enter」他就會開始執行了

以往我們開發功能因為在同一個工作區，所以需要一個完成後才能執行下一個

但用 Git Worktree 的好處，就是可以多個 AI Agent 同時執行

這邊我也把另外兩個 Worktree 打開，輸入「/」讓他們根據 workflow 來執行

但知道如果如果要在多個視窗切換，來觀察不同 AI Agent 執行的進度有點麻煩

所以我們可以點擊上方的 Agent Manager，這樣就能知道 Agent 執行的過程是否需要我們介入

像這邊轉圈圈就是代表執行中，而稍等一段時間後，就會變成鈴鐺提醒的符號

稍等一段時間後，AI 就在不同的工作區，完成了各自的功能，這邊我們也可以從 SPEC 的修改中，看到他已經自我驗證了功能符合需求

但 AI 完成後，人類一定也要自行確認結果是否符合期待，這邊我先開啟各個 worktree 的瀏覽器

把所有服務都啟動後，也剛好趁這個機會，跟大家說明不同功能分 feature 開發的好處

首先是我們人類驗證功能會比較輕鬆，因為要確認的面向單一

- 像 Cookie 的部分，就是下方在使用者進入時，會有彈窗提示
- 而淺色深色主題這塊，則是上方多了一個按鈕，讓使用者自由切換主題
- 至於常見問答，也有根據主題新增了對應的區塊

再確認修改符合預期後，先把 Agent Manager 關閉，我們要把 Worktree 的變更推送上去

比如我們來打開淺色深色主題的 Worktree，以往我們可能是人工手打，或是點擊「Generate」讓 AI 自動產生

但人工手打不僅耗時，每個人的撰寫習慣也各不相同；但假使點擊 Generate 讓 AI 自動產生，他生成的訊息有時候是英文，有時是中文，而且長短很隨機

所以 commit 這塊，我也有設計一個 Skill，讓 AI 根據我的指引生成

我們先回到專案的目錄，點擊「.agent」資料夾，在 skills 底下有一個「git-smart-commit」的技能

他的任務就是把雜亂的變更，拆分成有意義的 commit

畢竟請 AI 輔助開發時，很可能一口氣改了很多個檔案，如果全部放在一個 commit，那這個 commit 就跟沒寫一樣

這邊我們把左側的選單切換到 Source Control，你就會看到這個淺色深色主題有修改到多個檔案

這邊我就在對話窗輸入
```
新增 commit
```

稍等一段時間，會看到 AI 根據變更的檔案，告訴我們應該設計 2 個 commit 來分段上傳

如果覺得沒問題，就直接讓 AI 執行即可

執行完畢後，左側變更的檔案就消失了，記得要點擊 Publish Branch 才會將檔案上傳喔

完成後可以把這個工作區關閉，另外兩個 Worktree 也是一樣操作即可

> 因為範例比較簡單，所以剛好都是一個 commit 解決，所以我這邊補充一個情境
> 如果今天任務是「生成多國語系」，就會動到多個檔案，此時請 AI 生成 commit 時，他就可能會判斷不同檔案要分段 commit

都 Publish 之後，我們回到 GitHub 網頁，此時你會看到下拉選單中有剛剛上傳的 branch

隨便點擊一個，就可以看到剛剛撰寫的 commit 訊息

但目前的 feature 都在不同的 branch，所以接下來會帶大家透過 Pull Reqeust，將功能合併到主線

## ⑤ 好的 Pull Request，讓 Code Review 變輕鬆

這邊先點擊上方的「Pull Requests」，選擇「New Pull Requests」

如果你是 fork 我的專案，這邊要特別注意，base 要改成自己的帳號，然後 compare 要選擇想合併進來的 feature 分支

你會看到要填寫標題與描述，根據我的工作經驗，PR 是決定專案品質的重要環節

因為他是讓團隊成員 Code Review 時，理解你到底做了些什麼的文件

如果隨便寫寫，那審核的人通常也是隨意看看

畢竟直接讀程式碼，去理解功能為何這樣設計，真的太消耗腦力了

我知道很多人嫌寫 PR 很麻煩，所以我也設計了一個 Skill，用來生成 Pull Request 的模板

這邊我們把視角切換到 Antigravity，這個 Skill 一樣是放在「.agent/skills」底下，名稱叫做「git-pr-description」

這邊跟大家簡述這個 Skill 的設計

1. 首先，預設合併的目標為 master 這個主線分支，如果你是用 main/develop 等其他的分支請自行調整
2. 接下來就會把當前的分支與目標分支進行比對，讀取差異的檔案與 commit 訊息，這邊也再次跟大家提醒 commit 訊息是很重要的，你不想寫，至少要讓 AI 幫你寫吧
3. 而後就是分析變更的內容，然後生成 Pull Request 的 Title 與 Description
4. 其中 Description 的部分，我是要求他參考「pr-template」的這個範例檔來撰寫，如果大家想把這個 Skill 導入目前的工作流，可以改寫這個檔案

講了這麼多，我直接示範給大家看比較快

一樣打開 Agent Manager，然後再各自的 Agent 輸入

```
撰寫 PR
```

稍等一段時間後，他就會根據 Skill 提供的格式生成 Pull Request 的草稿了

這邊我們把它複製到 GitHub 後，我強烈建議大家至少要跟著下面的測試案例執行過一遍

如果是前端專案的話，建議附上對應的圖片，這樣可以減少別人 Code Review 時的負擔，同時也確認了功能符合預期

如果專案是自己開發的，那就直接點擊 Merge 合併就好

接下來我們再回到 Antigravity，合併好的 Worktree，我個人會建議直接點擊右鍵，在 Worktree Action 當中選擇 Remove Worktree 將他移除，避免 Worktree 太多管理困難

至於另外兩個 Worktree 也是一樣操作
先建立 Pull Request，接著用 AI 生成對應的 Review 草稿，確認沒問題後合併回主線，完成這一切後記得移除 Worktree

但有一點要提醒一下大家，因為 Worktree AI Agent 同時開發，所以開發當下不會知道彼此的變更

所以把功能合併回主線時，可能出現衝突；像常見問答跟淺色深色的分支在合併時，都遇到了這個問題

點擊「Resolve conflicts」可以看到發生衝突的位置，但我個人並不建議在網頁上調整，因為你無法得知調整後的實際狀況

我建議回到程式碼編輯器上操作，本地解決完衝突，確認結果符合預期後再更新上去

這邊先回到 Antigravity 的頁面，我以切換淺色深色主題這個分支來做示範

首先，我們要打開終端機，執行 `git fetch origin` 拉取遠端最新資料
然後再執行 `git merge origin/master` 來同步主線的更新

執行後，你就會發現左側 Source Control 這個地方，在 Merge Changes 有一個衝突的檔案

邏輯衝突較難處理，因為要考量到相容性或是改寫程式，這個之後有機會透過一個主題專門分享

而現在看到的這個是調整 CSS 網頁外型的語法衝突

只是括號位置撞到了而已，所以我選擇 Accept Current Change

這邊也跟大家提一下前面安裝 Gitlens 外掛的好處，像這邊滑鼠移到程式碼時，你就可以看到當時是誰更新了這段程式、更新的目的是什麼

好的，處理完衝突後，按下「+」，然後輸入 commit

這是與主線同步的意思

然後點擊「Sync Changes」推送上去

最後，我再跟大家補充 Git Worktree 會遇到的問題

除了前面跟大家展示的合併衝突外，還有可能出現「重複造輪子」的狀況

因為各分支各自開發，很可能各自實作了功能相似的程式；如果看都不看就合併，專案會多一堆冗贅程式。


雖然這個方案可以讓 AI Agent 同時開發，但因為每個 Worktree 通常對應不同分支，彼此的變更不會自動同步，所以把功能合併回主線時可能會出現以下狀況：
1. 重複造輪子
    首先，因為各分支各自開發，可能在不知情的情況下實作了功能相似的程式；如果看都不看就合併，專案會多一堆冗贅程式。
2. 程式碼衝突
    另外，如果 Worktree 同時修改到一段程式，合併時就會產生衝突，此時你就需要手動修正。

所以在使用上，我會建議大家：
1. 若涉及共用功能，請優先開發並先合併到主分支，作為後續開發的基礎
2. 並且當主分支有變更時，其他 Worktree 要先同步後再繼續開發，避免差異過大
3. 然後發 Pull Request 前，建議先將主分支的變更整合到當前分支，若有衝突先在本地解決
