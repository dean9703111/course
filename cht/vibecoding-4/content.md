# 痛點篇：為什麼需要規格驅動開發

> 少了文件，沒人知道 AI 寫程式的過程發生了什麼事，未來也難以維護。

---

## 業界的真實痛點

### 😓 企業導入 AI 開發後遇到的問題

- **需求落差**：不管前期規劃多完美，上線後還是收到一堆需求變更
- **專案越改越亂**：AI 寫程式越來越快，但缺乏規格引導，專案越改越壞
- **知識斷層**：關鍵人物離職，沒有文件，系統知識直接消失

> **開發系統的人不用系統，會用系統的人不會開發**
> 以人力資源系統來舉例，雖然是工程師開發，但 HR 才是主要的使用者。不管前期規劃與設計有多麼詳盡，初步完成的產品在進入易用性測試後，還是會收到一大堆需求變更。
>
> 會有這樣的結果，並不是使用者很刁鑽；而是真正的問題，往往在使用後才會被發現。

### 🔄 Vibe Coding 帶來的改變

- 有些企業想導入 Vibe Coding，目標是讓熟悉業務的人有能力設計出符合自己需求的產品
- 完成的產品原型，未來可以交給工程師做後續的優化與維護
- AI 已經可以讓沒有任何程式基礎的人，生成出一套涵蓋前後端與資料庫的系統

> **OpenSpec 解決了什麼？**
> 透過與 OpenSpec 的搭配，讓 AI 幫忙撰寫專案文件，達成用規格驅動開發的目標。同時解決過去小白不知道如何撰寫規格，以及工程師不想寫文件的痛點。
>
> 在 OpenSpec 的幫助下，我們不僅能完成專案從 0 到 1 的建置，更可以做到從 1 到 100 的迭代。

## 缺乏版控概念，是災難的起點

### 💥 經典錯誤案例

想像你跟同事共用一份 Google 試算表，你改了 A 欄、他改了 B 欄，結果兩人同時覆蓋了對方的修改。版本控制就是用來解決這類問題的 —— 它會幫每個人的修改留下紀錄，合併時也能看出誰改了什麼。

在團隊合作時，如果想把自己的修改合併回主線，會先開一個 Pull Request（簡稱 PR，你可以想成是一份「我改了這些東西，請幫我看一下」的申請單），讓其他人幫你檢查程式碼、討論是否需要修改，確認沒問題後再合併。

- **一個分支塞多個功能**：標題寫著「新增 i18n 多語系、淺色深色主題設計、常見 QA 問答區」等多個功能；從專案管理的角度，每個功能都應該對應到不同的 feature branch（功能分支，就像在 Word 裡另存一個副本來改，改好再合回去）
- **Files changed 難以追蹤**：涵蓋了 18 個檔案，有近千多行的程式變更；從 Code Review（讓同事幫你檢查程式碼有沒有問題的流程）的角度，你根本無法了解變更的功能要對應到哪些檔案
- **只有一個 Commit**：改了這麼多功能，竟然只有一個 commit（存檔紀錄，每次 commit 就是替專案拍一張快照），而且訊息也非常簡單；未來發生問題時，無法追溯當時的設計邏輯

> **這些問題在使用 AI 加速開發後，被成倍放大**
> AI 產生的程式碼很多，但這不是你不檢查的原因。我們應該去想怎麼利用 AI 去優化檢查流程。
>
> 別忘記，當你對 AI 產出的內容按下同意的那一刻，所有的錯誤必須由人類來承擔。

### 📋 好的版本控制應該做到

[flow]
1. 不同功能使用不同的 feature branch — 以減少 Code Review 的認知負擔
2. 根據調整檔案分階段 commit — 讓邏輯在未來可以被追蹤
3. 撰寫 Pull Request 草稿 — 讓團隊協作的資訊有一致性
[/flow]

## 確認 Node.js 版本

### 📦 安裝 Node.js

Node.js 是讓 JavaScript 程式語言可以在電腦上執行的環境，很多開發工具都建構在它之上。因為今天分享的 OpenSpec 與實作的全端專案會使用到 Node.js，所以要先確認電腦是否有安裝。

[flow]
1. 點擊上方的 Toggle Panel 打開終端機
2. 輸入 `node -v` 確認是否已安裝
3. 如果沒有顯示版號，需要先安裝 Node.js
[/flow]

> **建議透過 NVM 來安裝 Node.js**
> Node.js 有多個版本，NVM 這款工具可以幫你快速在不同版本間切換、更新。

**Mac** 安裝指令：

```prompt [label="安裝 NVM（Mac）"]
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

**Windows** 使用者請到 [nvm-windows](https://github.com/coreybutler/nvm-windows/releases) 下載 exe 檔安裝。

> **安裝完 NVM 後，請關閉終端機再重新開啟**
> NVM 安裝時會修改系統設定，必須重開終端機才會生效。如果輸入 `nvm` 顯示找不到指令，通常就是還沒重開。

安裝完成後：

```prompt [label="安裝 Node.js"]
nvm install --lts
```

輸入 `node -v` 便可檢視安裝的版本。

## 安裝 OpenSpec

### 🧩 全局安裝 OpenSpec

在 Node.js 安裝完成後，在終端機輸入指令便可完成全局安裝。


- 只要用白話的方式與 AI 對話，就能建立完善的規格

- 可以盡量避免過去用 AI 寫程式時，相同功能重複被寫好幾次的痛點

# 實戰篇：OpenSpec 規格驅動開發全端系統

> 用規格驅動開發，從零打造涵蓋前後端與資料庫的全端系統。
>
> **什麼是全端？** 前端是使用者看到的網頁畫面與操作介面；後端是在背後處理資料的程式邏輯；資料庫則是儲存所有資料的地方（例如員工的打卡紀錄）。三者合在一起，就是一個完整的「全端」系統。

---

## 安裝 Docker

### 🐳 為什麼需要 Docker

今天會帶大家完成一個全端專案。根據過去經驗，最容易在環境設定上卡關的是「資料庫」——資料庫就像一個大型的 Excel 檔案，專門用來儲存系統的所有資料，但它的安裝與設定比 Excel 複雜很多。

- 你可以把 Docker 想成一個「標準化的容器」，把專案需要的環境通通打包在裡面
- 不管你用 Mac 還是 Windows，打開就能跑，不用自己一個一個裝
- 也方便未來與其他人一起維護專案

到 [Docker 官網](https://www.docker.com/) 根據自己的作業系統下載安裝後，就可以直接使用，不需要特別註冊、登入帳號。

> **安裝完成後，記得啟動 Docker Desktop**
> Docker 安裝完並不會自動啟動。請在應用程式中找到 Docker Desktop 並打開，**註冊相關步驟都可以直接 Skip**，看到左下角顯示綠色的 Running 狀態，才代表 Docker 已經準備好了。

## 設計提示詞

### 📝 Prompt 設計架構

以大部分上班族都會使用到的「出缺勤系統」作為示範範例。

```prompt [label="出缺勤系統提示詞"]
請設計一個中文的出缺勤管理系統，員工可以透過此系統打卡上下班
權限分成管理者與一般使用者，管理者可以修改、新增/刪除使用者
新增使用者時，會發送 Email 給予登入密碼
員工可透過系統請假、申請加班，申請假單需要簽核流程，並且有年假、補休申請，代理人制度

前端使用 React，後端使用 Express，資料庫用 Postgres
希望透過 Docker 啟動，並且要包含 Postgres Admin 網頁
這是初步需求，我們可以透過討論釐清細節後，參考 openspec 的 skill 執行
```

### 💡 提示詞三大區塊

- **專案目標**：大概說明需求，一開始描述有點模糊沒關係，AI 會在後續跟你釐清細節
- **使用技術**：如果希望專案完成後能由工程師團隊接手維護，一定要先詢問他們最擅長的前端、後端、資料庫技術
- **細節討論**：提醒 AI 要與我們討論規格的細節，釐清一開始沒說清楚的地方，並要求他參考 OpenSpec 的 Skill

> **使用 Plan Mode 讓計劃更詳細**
> 開啟對話視窗，把對話模式切換成 Plan Mode 可以讓 AI 可以更仔細規劃
> 但如果想加速體驗，可以在提示詞最後加上「以最小可行性方案來規劃」。

### 🤖 與 AI 討論細節

- AI 會根據需求提出問題，讓細節更加完善
- 根據實際需求回答即可
- 在 Plan 模式下，會先設計出一份實作規劃書
- 確認內容符合預期後，便可執行後續的實作

## OpenSpec 撰寫專案文件

### 📄 OpenSpec 的文件產生流程

釐清完需求後，AI 會觸發 OpenSpec 的 Skills 做更細部的規劃。相關的規格文件都會存放在 `changes` 資料夾底下。

[flow]
1. proposal.md — 確認目標是否正確、範圍是否合理（確認後才會進入下一步）
2. design.md — 說明技術挑選的理由，以及專案可能會遇到的風險與處理方案
3. specs 資料夾 — 根據功能區分資料夾，分別建立說明細節的規格文件
4. task.md — 用來檢驗任務是否完成的列表，目前都是空的，等任務完成後才會打勾
[/flow]

> **如果文件內容不符合預期怎麼辦？**
> 直接告訴 AI 哪裡需要調整即可，例如「proposal 的範圍太大了，我只需要打卡功能」。AI 會根據你的回饋重新修正，你只要擔任監督的角色即可。

## 規格驅動開發（SDD）

### ⚡ 開始實作

確認規劃都符合預期後，告訴 AI「開始實作」，AI 就會根據規劃開始撰寫程式。

完成後，Task 的任務列表會全部打勾。如果不知道如何啟動專案，可以請 AI 幫忙：

```prompt [label="啟動專案"]
幫我啟動專案，我想在 Local 開發前後端程式
只有 Postgres 跟 Postgres Admin 用 Docker 啟動
如果根目錄 README.md 沒有操作步驟，請協助補充
```

> **建議第一次啟動請 AI 幫忙**
> 處理較大型的專案時，AI 有很高的機率在第一版遇到一些零星錯誤，我們可以先讓他自我修正一下。

### 🏗️ 專案架構說明

- **frontend 資料夾**：前端程式，就是你看到的網頁畫面跟操作邏輯
- **backend 資料夾**：後端程式，負責資料處理、跟資料庫溝通
- **`.env` 環境檔**：存放資料庫帳號密碼、Email SMTP 設定等隱私資訊
- **docker-compose.yml**：Postgres 資料庫與 pgAdmin 都是透過 Docker 啟動
- **README.md**：專案簡介以及 Local 開發的步驟

## 初步驗證功能

### 🧪 確認功能是否符合預期

- 可以正常打卡，打卡時間不在上班時間內時，會有遲到的訊息
- 管理者可以進行使用者的新增、權限的設定
- 有請假、加班申請的頁面
- 管理者可以自由設定上下班的時間區間

### 🗄️ 查看 Postgres Admin 資料庫

確認網頁內容可以順利瀏覽後，可以打開 Postgres Admin 網頁，看一下資料庫是否有順利寫入。

- 登入後需要註冊 Server
- 連線資訊通常放在後端的 `.env` 環境檔中
- 如果不知道資訊如何填寫，直接問 AI 就好

## 接受變更並建立專案規則

### ✅ 歸檔此次變更

初步確認功能符合預期後，告訴 AI 進行歸檔：

```prompt [label="歸檔"]
功能符合預期，進行歸檔
```

- 原本空空的 `specs` 資料夾底下，會出現專案對應的規格文件
- `archives` 底下出現歸檔的文件，並註明對應的日期

### 📌 建立專案規則

在完成專案初步建立後，在 AI 對話窗輸入下面要求：
```prompt [label="撰寫專案的 Rule"]
請根據目前專案的實際檔案結構、目錄命名、設定檔、相依套件、程式碼分層方式與既有實作慣例，幫我產出一份「專案常駐上下文文件」。

這份文件的用途是：
- 在未來每次 AI 對話開始時自動載入
- 幫助 AI 快速理解這個專案的架構、技術選型、開發規則與修改原則
- 降低每次都要重新解釋專案的成本
- 提升後續開發、維護、除錯、重構時的準確度

請特別遵守以下原則：

【核心原則】
1. 只保留高價值、會在未來反覆使用的資訊
2. 內容要精簡，避免冗長，因為這份文件會被頻繁載入
3. 不要寫成教學文件，而要寫成 AI 協作時的專案記憶文件
4. 不要臆測不存在的規則；不確定的地方請標註「待確認」
5. 優先根據實際程式碼與設定檔推斷，不要只看檔名
6. 若發現專案已有明確慣例，請整理成可執行的規則
7. 若規則彼此衝突，請明確指出
8. 若資訊太細、太短期、太容易變動，請不要放進這份文件

【希望文件涵蓋的內容】
1. 專案用途與主要目標（簡短）
2. 技術棧與主要依賴
3. 目錄結構與模組責任分工
4. 前後端/資料層/API/共用模組之間的關係
5. 關鍵開發規則與既有慣例
6. 命名風格、檔案組織、元件/模組拆分原則
7. 狀態管理、資料流、錯誤處理、驗證方式
8. 環境設定、重要指令、啟動/測試/建置方式
9. AI 在這個專案中協作時應遵守的修改原則
10. 明顯的風險區域、歷史包袱、不可隨意變動的部分
11. 待確認事項（若有）

【輸出要求】
- 請輸出為 Markdown，放在「.agents/rules/project.md」
- 以「給 AI 使用的專案上下文文件」風格撰寫
- 優先短句、條列、明確規則
- 避免空泛描述，例如「請保持良好程式碼品質」
- 規則要具體，例如「新增 API 呼叫請集中於 xxx 目錄，不要直接寫在元件內」
- 若內容過長，請主動壓縮成高密度版本

【輸出格式】
請使用以下結構：

# Project Context

## 1. Project Overview
## 2. Tech Stack
## 3. Architecture
## 4. Directory Responsibilities
## 5. Development Conventions
## 6. Data Flow / State / API Rules
## 7. Risky / Sensitive Areas
```

- 這份文件會在每次對話開始時自動載入
- 幫助 AI 在未來開發過程中，更了解這個專案的架構、技術、開發規則

### 📐 建立 OpenSpec 設定

```prompt [label="填寫 OpenSpec 設定"]
Please read openspec/config.yaml and help me fill it out
with details about my project, tech stack, and conventions
```

- 這是請 AI 幫你填寫專案使用到的技術，以及需要遵守的規則

> rules 底下的 `project.md` 是給「做事」用的，`openspec/config.yaml` 是給「規劃」用的。

# 版控篇：建立版本控制，根據需求迭代功能

> 每個指令都可能大幅調整專案程式，但 AI 修改的結果未必是你要的。版本控制是你的安全網。

---

## 為什麼需要版本控制

### 🛡️ Git 是你的保險

Vibe Coding 時，你的每個指令都可能大幅調整專案程式。但 AI 修改的結果未必是你要的，甚至有時會把正常的功能給改壞了。

- 安裝 [Git](https://git-scm.com/) 這個程式碼版本控制工具
- 註冊 [GitHub](https://github.com/) 帳號來管理專案
- 如果你是第一次接觸，可以參考[過去的教學影片](https://www.youtube.com/watch?v=5DEV_K0JauQ&t=836s)

## 設計 .gitignore

### 🚫 哪些檔案不該上傳

版本控制並不代表需要把專案底下所有檔案都上傳。

```prompt [label="設計 .gitignore"]
請幫我設計專案的「.gitignore」
但「openspec」要加入版本控制
```

- **環境變數檔案**（`.env`）：就像一張寫著帳號密碼的小抄，只放在自己電腦上，不能上傳
- **套件資料夾**（`node_modules`）：存放自動安裝的套件，不需上傳
- **編譯結果**（`dist`）：自動產生的結果，不需上傳
- **系統暫存檔、日誌**：不需要加入版控

## 建立版本控制

### 📤 切換分支並上傳變更

我們要先切換到 `develop` 分支，避免直接在主分支上開發。

[flow]
1. 點擊左下角的分支名稱（通常顯示 main）
2. 選擇 Create new branch，輸入 `develop`
3. 點擊左側的 Source Control
4. 點擊 Generate，AI 會辨識更新內容來產生 commit 訊息
5. 確認文字描述符合預期後，點擊 commit 記錄變更
6. 點擊 Sync Changes 正式上傳到 GitHub
[/flow]

## 迭代功能

### 🔄 從 1 到 100 的迭代

不管 Vibe Coding 還是開發系統，最難的並不是 0 到 1，而是 1 到 100 的那段路。但有了 OpenSpec 的幫助，這一切會變得容易許多。

```prompt [label="新增忘打卡功能"]
幫我設計忘打卡的處理方案，讓規則彈性一點，使用 OpenSpec
```

- AI 會開始著手設計，並根據 OpenSpec 的 Skill 建立對應的需求規格
- 由於有加入 Git 做版本控制，可以在 Source Control 中清楚看到 AI 撰寫的需求變更規格書
- 確認需求與期待相符後，直接說「開始實作」就好

## 驗證新功能

### 🧪 確認迭代結果

因為功能涉及後端與資料庫，建議先請 AI 確認：

```prompt [label="重啟確認"]
請協助確認資料庫變更是否執行，並幫我重新啟動後端程式
```

### 🐛 遇到問題就直接討論

- 刷新網頁後找不到忘打卡功能？直接跟 AI 說「我找不到使用忘打卡的地方」
- 送出申請失敗？直接描述遇到的錯誤訊息
- 想調整規則？直接提出需求

```prompt [label="調整規則"]
我希望忘打卡可以覆蓋既有的紀錄
```

> **AI 完成後，人類一定要自行確認結果是否符合預期**
> 不管 AI 有多強大，出事還是要人類來背鍋。

### ✅ 將變更歸檔

確認沒問題後：

```prompt [label="歸檔"]
幫我歸檔
```

- `archive` 裡面會多一個資料夾，放的就是這次的變更
- 專案的 `specs` 也會進行對應的調整——在左側 Source Control 面板中，綠色代表新增的檔案，黃色代表有修改的檔案

### 🌿 設計 Branch name

```prompt [label="新增分支"]
分析目前的變更內容，依照 camelCase 命名規則生成 feature branch 名稱，並建立與 checkout 該 branch
```

> **Tips**
> 不是所有事情都需要觸發 OpenSpec，像 branch 命名直接讓 AI 想就好

# 協作篇：設計 Pull Request

> 拆分 Commit 讓歷史可以被追蹤，好的 Pull Request 讓 Code Review 變輕鬆。

## 拆分 Commit

### 📦 為什麼要拆分 Commit

請 AI 輔助開發時，很可能一口氣改了很多個檔案。如果全部放在一個 commit，那這個 commit 就跟沒寫一樣。

- 人工手打不僅耗時，每個人的撰寫習慣也各不相同
- 讓 AI 自動產生，生成的訊息有時候英文、有時中文，長短很隨機
- 專案中有設計一個 `git-smart-commit` 技能，讓 AI 根據指引生成

### ✍️ 執行智慧 Commit

在 Source Control 確認有哪些檔案被修改後，在對話窗輸入：

```prompt [label="產生 Commit"]
新增 commit
```

- AI 會根據變更的檔案，告訴你應該設計幾個 commit 來分段上傳
- 確認沒問題後，讓 AI 執行即可
- 完成後記得點擊 Publish Branch 將檔案上傳

---

## 生成 Pull Request

### 📋 為什麼 PR 很重要

PR 是決定專案品質的重要環節，因為他是讓團隊成員 Code Review 時，理解你到底做了些什麼的文件。

- 如果隨便寫寫，審核的人通常也是隨意看看
- 直接讀程式碼去理解功能為何這樣設計，真的太消耗腦力了
- 專案中有設計一個 `git-pr-description` 技能，用來生成 Pull Request 的模板

### 🔍 PR 生成 Skill 的運作邏輯

[flow]
1. 預設合併目標為 master 主線分支（如果用 main/develop 請自行調整）
2. 比對當前分支與目標分支，讀取差異的檔案與 commit 訊息
3. 分析變更內容，生成 PR 的 Title 與 Description
4. Description 會參考 `pr-template` 的範例檔來撰寫
[/flow]

### 🚀 建立 Pull Request

```prompt [label="生成 PR"]
撰寫 PR，目標合併到 develop
```

- AI 會根據 Skill 提供的格式生成 Pull Request 的草稿
- 將草稿複製到 GitHub 後，強烈建議至少跟著測試案例執行過一遍
- 如果是前端專案，建議附上對應的圖片，減少別人 Code Review 時的負擔

### ⚠️ 合併時可能遇到衝突

因為不同功能同時開發，所以開發當下不會知道彼此的變更。把功能合併回主線時，可能出現衝突。

- 建議回到程式碼編輯器上操作，本地解決完衝突，確認結果符合預期後再更新上去
- 不建議在 GitHub 網頁上直接調整，因為你無法得知調整後的實際狀況

### 🔧 本地解決衝突的步驟

[flow]
1. 在終端機執行 `git fetch origin` 拉取遠端最新資料
2. 執行 `git merge origin/master` 同步主線的更新
3. 左側 Source Control 會出現衝突的檔案
4. 打開衝突檔案，選擇要保留的版本
5. 點擊 Sync Changes 推送上去
[/flow]

> **衝突畫面怎麼看？**
> 打開衝突檔案時，編輯器會用不同顏色標示兩個版本，並提供選項讓你決定保留哪一個：
> - **Accept Current Change**（保留你自己的版本）
> - **Accept Incoming Change**（保留別人的版本）
> - **Accept Both Changes**（兩邊都保留，再自行調整）
>
> 如果看到 `<<<<<<< HEAD` 這樣的標記不用慌，這就是 Git 標示衝突的方式，選完版本後標記會自動消失。

> **都合併完成後**
> 在 develop 分支輸入 `git pull` 拉回變更，然後啟動專案確認合併後的內容是否符合預期。

# 總結：AI 時代下，每個人正在經歷角色的轉變

> 市場需要好產品，而好產品需要「態度」。

---

## 角色的轉變

### 🎯 AI 正在改變企業決策

- 以往出缺勤管理系統，企業都是找特定廠商購買，並支付年費來維護
- Vibe Coding 的出現，可能會導致企業做出不同的選擇
- 過去程式碼是很昂貴的產品，不是程式碼本身貴，而是產生有意義的程式需要大量的經驗與專業
- AI 的發展正在讓程式碼快速貶值，因為 AI 寫的程式不只快，品質還越來越優秀

> **有一點從過去到現在都沒有改變**
> 市場需要的是好產品。而想打造出一個好產品，現在態度會更重要。

### 🔄 每個人的角色都在轉變

- **工程師**：重點不再是寫了多少行程式，而是去評估系統架構、審查程式碼合理性
- **PM 與執行業務單位**：不再只是眼巴巴的等工程師開發，在 AI 的輔助下已經可以嘗試加入開發

> **不管 AI 有多強大，出事還是要人類來背鍋**
> 當你按下同意的那一刻，責任就歸屬於你。
